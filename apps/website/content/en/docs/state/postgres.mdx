# Postgres State

Persist approval requests in Postgres for production deploys when you already run a Postgres database. Pass `PostgresState` as the `state` option in `new Hitl()`.

## Install

```bash
npm i @hitl-sdk/hitl @hitl-sdk/state-pg pg
```

## Run Postgres migrations

Run migrations before deploy or on first boot. Default table: `hitl.human_requests`.

```bash
export HITL_POSTGRES_URL=postgres://user:password@host:5432/database
npx @hitl-sdk/state-pg setup
```

`DATABASE_URL` is also accepted. Prefer `HITL_POSTGRES_URL` to keep Hitl persistence explicit.

Export DDL without connecting:

```bash
npx @hitl-sdk/state-pg schema
```

## Set up the Hitl server with Postgres

```typescript
// lib/hitl.ts
import pg from "pg";
import { Hitl } from "@hitl-sdk/hitl";
import { PostgresState } from "@hitl-sdk/state-pg";
import { workflowResolver } from "@hitl-sdk/resolver-workflow-sdk";

const pool = new pg.Pool({
  connectionString: process.env.HITL_POSTGRES_URL ?? process.env.DATABASE_URL,
});

const state = new PostgresState(pool);
await state.ensureSchema(); // idempotent; or rely on `setup` at deploy time

export const hitl = new Hitl({
  state,
  resolver: workflowResolver(),
});
```

`PostgresState` accepts any `PgQueryable` pool. Neon, Supabase, or `pg-mem` work as long as they speak Postgres SQL.

[Vercel Workflow SDK worlds](https://workflow-sdk.dev/worlds) use their own storage and migrations. This package only manages Hitl tables.

## Migrations

Re-run `setup` or `ensureSchema()` after upgrading `@hitl-sdk/state-pg` to apply new migrations idempotently.
