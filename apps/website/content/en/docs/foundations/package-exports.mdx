# Package exports

`@hitl-sdk/hitl` publishes multiple entry points. Import from the subpath that matches your layer.

## Subpath map

| Export path | Key symbols | Use for |
|-------------|-------------|---------|
| [`@hitl-sdk/hitl`](#main-entry) | `Hitl`, `actions`, `field`, `remind`, `escalate`, `isResolved` | Server setup + workflow building blocks |
| [`@hitl-sdk/hitl/client`](#client) | `createHitlClient`, `HitlClient`, option types | Engine binding authors |
| [`@hitl-sdk/hitl/core`](#core) | `createHumanRequest`, `HitlRuntime`, inbox services | Advanced server integration |
| [`@hitl-sdk/hitl/state`](#state) | `InMemoryState`, state record types | Dev and custom state backends |
| [`@hitl-sdk/hitl/adapter`](#adapter) | `HitlAdapter`, delivery types | Custom channel adapters |
| [`@hitl-sdk/hitl/testing`](#testing) | `createTestHitl` | Unit and integration tests |

Workflow functions (`requestHuman`, `waitForHuman`, `notify`) are **not** on the main entry. They come from your engine-bound client. See [Foundations overview](/docs/foundations/overview).

## Client

`import { createHitlClient } from "@hitl-sdk/hitl/client"`

Low-level workflow-side HTTP client. Engine packages (`@hitl-sdk/resolver-workflow-sdk`, `@hitl-sdk/resolver-temporal`, `@hitl-sdk/resolver-inngest`) wrap this. Most apps use the engine wrapper instead.

| Symbol | Description |
|--------|-------------|
| `createHitlClient` | Build a `HitlClient` from engine primitives |
| `HitlClient` | `requestHuman`, `waitForHuman`, `notify` |
| `HumanPending`, `HumanBatchPending` | Pending handles from create phase |
| `RequestHumanOptions`, `WaitForHumanOptions`, `HumanWaitOptions` | Option types |
| `HumanItem` | Batch item shape |
| `NotifyOptions` | Notify call options |
| `CreateHitlClientOptions` | `url`, `basePath`, `secret`, `suspend`, `sleep`, `request` |
| `DEFAULT_BASE_PATH` | `"/.well-known/hitl/v1"` |

See [waitForHuman](/docs/foundations/wait-for-human), [requestHuman](/docs/foundations/request-human), [notify](/docs/foundations/notify).

## Core

`import { … } from "@hitl-sdk/hitl/core"`

Runtime services used by `Hitl` and advanced integrations. Includes `createHumanRequest`, `createBatchRequest`, `notifyVia`, `remindHumanRequest`, `timeoutHumanRequest`, and inbox resolution helpers.

Use when building custom server middleware or extending delivery behavior. Most apps only need `new Hitl({ … })` from the main entry.

## State

`import { InMemoryState, … } from "@hitl-sdk/hitl/state"`

In-memory state for local development. Production apps typically use `@hitl-sdk/state-sqlite`, `@hitl-sdk/state-pg`, or `@hitl-sdk/state-ioredis`.

See [State](/docs/state/sqlite) docs for persistence options.

## Adapter

`import type { HitlAdapter, … } from "@hitl-sdk/hitl/adapter"`

Types and helpers for building custom channel adapters. See [Chat SDK adapter](/docs/channels/chat-sdk) for the bundled Slack/Teams/Discord integration.

## Testing

`import { createTestHitl } from "@hitl-sdk/hitl/testing"`

Spin up an in-process Hitl server for tests. No HTTP mount required.
